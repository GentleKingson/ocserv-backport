# Build the ocserv backport on Debian 13 trixie

This guide explains how to build local backport packages for `ocserv 1.5.0`
on a Debian 13 trixie build host. Prefer the automatic build path. Use the
manual build sections only when the environment must be prepared manually.

The default local package version generated by the repository is
`1.5.0-1~debian13.1`. To change the version, override it with the
`OCSERV_VERSION` environment variable.

## Automatic build prerequisites

Before using the automatic build, prepare:

- A Debian 13 trixie build host.
- A current user with root or sudo privileges.
- Network access to the Debian mirror, Docker official APT repository, Docker
  registry, and GitHub.
- A native `amd64` or `arm64` build host. `arm64` builds should run on a native
  arm64 Debian 13 host or runner.

When using `--provision`, `sbuild`, `schroot`, `lintian`, and Docker do not
need to be installed ahead of time; the script prepares them. When running
without arguments, these tools, the Debian source signature verification
keyring, and the matching trixie sbuild chroot must already be available.

## Automatic build

If the repository is not present yet, clone it and enter the repository:

```bash
git clone https://github.com/GentleKingson/ocserv-backport.git
cd ocserv-backport
```

If the environment is already prepared, run the default checks and build:

```bash
scripts/trixie-auto-build.sh
```

This mode does not install dependencies, configure Docker CE, or create an
sbuild chroot. It only checks the existing environment. After the checks pass,
the script continues with the full Debian trixie build.

If this is a new Debian 13 trixie build host, or if the default checks report
that provisioning is needed, run:

```bash
scripts/trixie-auto-build.sh --provision
```

`--provision` installs build dependencies, checks the Debian source signature
verification keyring, configures Docker CE, and creates the
`trixie-${TARGET_ARCH}-sbuild` sbuild chroot after confirmation.

For unattended environments, use:

```bash
scripts/trixie-auto-build.sh --provision --yes
```

`--yes` only auto-confirms chroot creation when the sbuild chroot is missing.
It does not bypass checks for existing directories, broken chroots, the
non-native architecture guard, or other safety checks.

If a normal user has just been added to the `sbuild` group, the script prompts
you to continue in a new shell. Enter the new shell as instructed, then rerun
the automatic build.

When `TARGET_ARCH` is unset, the Debian scripts auto-detect native `amd64` or
`arm64`. You can also set `TARGET_ARCH` explicitly to override the target
architecture. The aliases `x86_64 -> amd64` and `aarch64 -> arm64` are
supported. By default, only native architecture builds are supported. If the
explicit target architecture differs from the native architecture, the script
fails early.

Only set the following when you have already prepared an unsupported
non-native build environment manually:

```bash
ALLOW_NON_NATIVE_TARGET_ARCH=1 TARGET_ARCH=arm64 make trixie-build
```

`ALLOW_NON_NATIVE_TARGET_ARCH=1` only bypasses the native architecture guard.
It does not configure cross-build, QEMU, binfmt, or a foreign-architecture
chroot. This path is not supported by this project.

## Manual build prerequisites

When not using automatic `--provision`, prepare:

- A Debian 13 trixie build host.
- A current user with root or sudo privileges.
- Network access to the Debian mirror, Docker official APT repository, Docker
  registry, and GitHub.
- Working `sbuild`, `schroot`, `lintian`, and Docker CE installations.
- A created and usable `trixie-${TARGET_ARCH}-sbuild` sbuild chroot for the
  target architecture.
- A readable Debian source signature verification keyring.
- A normal build user with `sbuild` group membership. Running
  `make trixie-build` directly also requires access to the Docker daemon.

## Manually prepare the build host

The following steps apply only when not using
`scripts/trixie-auto-build.sh --provision`. If the automatic script has already
prepared the build host, do not repeat the chroot creation or user group setup
from this section.

Update the apt index:

```bash
sudo apt-get -q=1 -o=Dpkg::Use-Pty=0 update
```

Install build tools:

```bash
sudo apt-get -q=1 -o=Dpkg::Use-Pty=0 install -y --no-install-recommends \
  git ca-certificates curl gnupg \
  build-essential fakeroot devscripts dpkg-dev debhelper \
  debian-archive-keyring debian-keyring debian-maintainers \
  sbuild schroot debootstrap lintian libdistro-info-perl \
  python3 python3-yaml bats shellcheck make
```

Install Docker CE:

```bash
sudo apt-get -q=1 -o=Dpkg::Use-Pty=0 remove -y \
  docker.io docker-doc docker-compose docker-compose-v2 podman-docker \
  containerd runc || true
sudo apt-get -q=1 -o=Dpkg::Use-Pty=0 install -y --no-install-recommends ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/debian/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

cat <<EOF | sudo tee /etc/apt/sources.list.d/docker.sources >/dev/null
Types: deb
URIs: https://download.docker.com/linux/debian
Suites: trixie
Components: stable
Signed-By: /etc/apt/keyrings/docker.asc
Architectures: $(dpkg --print-architecture)
EOF

sudo apt-get -q=1 -o=Dpkg::Use-Pty=0 update
sudo apt-get -q=1 -o=Dpkg::Use-Pty=0 install -y \
  docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin
sudo systemctl enable --now docker
```

If the current system only has the root user, create a normal build user first:

```bash
adduser builder
usermod -aG sudo builder
su - builder
```

A normal user must be added to the `sbuild` group:

```bash
sudo sbuild-adduser "$USER"
newgrp sbuild
```

The default manual build runs `docker` directly. If the normal user must run
`docker info` or `docker run` directly, add the current user to the `docker`
group and log in again:

```bash
sudo usermod -aG docker "$USER"
```

Create the default `amd64` sbuild chroot:

```bash
sudo sbuild-createchroot \
  --arch=amd64 \
  --chroot-suffix=-sbuild \
  --include=eatmydata,ccache,gnupg,ca-certificates \
  trixie \
  /srv/chroot/trixie-amd64-sbuild \
  http://deb.debian.org/debian
```

Update the chroot:

```bash
sudo sbuild-update -udcar trixie-amd64-sbuild
```

Create the `arm64` sbuild chroot with the same Debian mirror:

```bash
sudo sbuild-createchroot \
  --arch=arm64 \
  --chroot-suffix=-sbuild \
  --include=eatmydata,ccache,gnupg,ca-certificates \
  trixie \
  /srv/chroot/trixie-arm64-sbuild \
  http://deb.debian.org/debian
```

Update the chroot:

```bash
sudo sbuild-update -udcar trixie-arm64-sbuild
```

## Version variables

The Debian scripts keep Debian source package identity separate from the local
backport version. The default local version comes from the `Makefile` and
Debian build scripts:

```text
OCSERV_VERSION=1.5.0-1~debian13.1

TARGET_SUITE=trixie
TARGET_ARCH=amd64  # Optional explicit override; auto-detected by the Debian script when unset
```

`OCSERV_VERSION` is used for `debian/changelog` and the final Debian trixie
build artifacts. Debian source package identity is defined by the locked
`ocserv 1.5.0-1` entry in `source-lock/`. Do not put the `~debian13.*` local
backport version in `source-lock` paths.

Override the local version with an environment variable:

```bash
OCSERV_VERSION=1.5.0-1~debian13.2 make trixie-build
```

If the build host uses non-standard Debian keyring paths, override the default
candidates with a colon-separated list:

```bash
DSCVERIFY_KEYRING_PATHS=/path/to/debian-keyring.gpg:/path/to/extra.gpg make trixie-build
```

## Manual build commands

Run the full build:

```bash
make trixie-build
```

Build a specific architecture:

```bash
TARGET_ARCH=arm64 make trixie-build
```

`TARGET_ARCH=arm64` only adjusts paths, artifact matching, and the `--arch`
argument passed to sbuild. The caller must already provide a usable native
arm64 build host and arm64 sbuild/schroot environment. The scripts do not
configure cross-build, QEMU, or binfmt.

To run the build in stages, execute the following targets in order:

```bash
make trixie-verify-locks
make trixie-fetch-ocserv
make trixie-rewrap-ocserv
make trixie-src-pkg-ocserv
make trixie-binary-ocserv
make trixie-lint
make trixie-smoke-basic
```

Run only the source package path:

```bash
make trixie-source-ci
```

`make trixie-source-ci` runs only:

```text
trixie-verify-locks -> trixie-fetch-ocserv -> trixie-rewrap-ocserv -> trixie-src-pkg-ocserv
```

It does not run the sbuild binary build, lintian, or the Docker smoke test.
Full `.deb` builds still require a native sbuild environment or the manual
Debian Trixie build workflow.

## Artifact directories

Debian trixie artifacts are isolated by architecture:

```text
build/debian/trixie/${TARGET_ARCH}/source/
build/debian/trixie/${TARGET_ARCH}/binary/
build/debian/trixie/${TARGET_ARCH}/keyrings/debian/
```

The final `ocserv` package is located in:

```text
build/debian/trixie/${TARGET_ARCH}/binary/
```

Source package artifacts are located in:

```text
build/debian/trixie/${TARGET_ARCH}/source/
```

When the automatic build refreshes the Debian source signature verification
keyring, it stores the temporary keyring in:

```text
build/debian/trixie/${TARGET_ARCH}/keyrings/debian/
```

## GitHub Actions

Pull request CI runs only static checks, lock verification, unit tests, and
stub orchestration tests.

Pull request CI does not create an sbuild chroot, run the Docker smoke test, or
build or upload binary `.deb` files. Changes limited to `docs/**` usually do
not trigger this PR CI; trigger the workflow manually when needed.

The `ci.yml` workflow verifies the source package build path on a weekly
schedule. It can also run manually with `target=source-package` or `target=all`.
This path runs in a `debian:trixie` container:

```text
trixie-verify-locks -> trixie-fetch-ocserv -> trixie-rewrap-ocserv -> trixie-src-pkg-ocserv
```

Source CI does not run `trixie-binary-ocserv`, `trixie-lint`, or
`trixie-smoke-basic`, and it does not upload GitHub Actions artifacts.

The manual workflow `.github/workflows/debian-trixie-build.yml` uses this
architecture matrix:

```text
amd64 -> ubuntu-24.04
arm64 -> ubuntu-24.04-arm
```

The fixed check names for the two jobs are `debian-trixie-build-amd64` and
`debian-trixie-build-arm64`. The workflow uses
`scripts/trixie-auto-build.sh --provision` on GitHub-hosted runners to prepare
the trixie sbuild environment, run the full Debian binary build, lintian, and
Docker smoke validation, and upload build artifacts.

On success, the artifact names are `debian-trixie-build-amd64` and
`debian-trixie-build-arm64`. The log artifact names are
`debian-trixie-build-logs-amd64` and `debian-trixie-build-logs-arm64`; the
workflow also attempts to upload them on failure.
